# 비트코인 직렬화 구현 계획

이 문서는 비트코인 네트워크의 블록, 트랜잭션, 그리고 SEC(압축, 비압축), DER, 공개키, 비밀키 등의 직렬화 및 역직렬화를 구현하기 위한 검토 사항들을 정리합니다.

## 1. 키 직렬화 (Key Serialization)

가장 먼저 개인키와 공개키를 표준 형식으로 변환하는 방법을 구현해야 합니다.

### 공개키 (SEC Format)

타원 곡선 위의 한 점 `(x, y)`를 바이트 시퀀스로 변환하는 표준입니다.

-   **비압축 (Uncompressed) 형식**:
    -   `0x04` 접두사 + 32바이트 `x` 좌표 + 32바이트 `y` 좌표 (총 65바이트)
-   **압축 (Compressed) 형식**:
    -   `y` 좌표가 짝수이면 `0x02`, 홀수이면 `0x03` 접두사 + 32바이트 `x` 좌표 (총 33바이트)
-   **검토 사항**:
    -   압축 형식을 구현하려면 `x` 좌표와 곡선 방정식(`y^2 = x^3 + 7`)을 이용해 `y` 좌표를 복원하는 로직이 필요합니다.
    -   이는 유한체 위에서 **모듈러 제곱근(modular square root)**을 계산하는 것과 같습니다. 현재 `FiniteField.hs`에 이 기능이 없으므로 추가 구현을 고려해야 합니다.

### 개인키 (WIF - Wallet Import Format)

개인키를 백업하거나 다른 지갑으로 가져올 때 사용하는 Base58Check 인코딩 형식입니다.

-   구조: `[버전 바이트] + [32바이트 개인키] + [압축 플래그(선택)] + [체크섬]`
-   **검토 사항**:
    -   **Base58Check 인코딩**: Base58 인코딩/디코딩 기능이 필요합니다. 체크섬은 `hash256`을 두 번 적용하여 계산하며, `ECDSA.Utils.Hash.hs`의 `hash256` 함수를 활용할 수 있습니다.
    -   Haskell에서는 `base58-bytestring` 같은 라이브러리를 `impl-btc.cabal` 파일에 추가하여 사용할 수 있습니다.

### 서명 (DER - Distinguished Encoding Rules)

`(r, s)` 서명 값을 ASN.1 DER 표준에 따라 인코딩하는 방식입니다.

-   구조: `0x30 [총 길이] 0x02 [r 길이] [r 값] 0x02 [s 길이] [s 값]`
-   **검토 사항**:
    -   이 형식은 매우 엄격하므로 바이트 하나하나를 정확히 다루어야 합니다.
    -   바이트 순서, 길이 표시, 정수 표현 방식(특히 첫 비트가 1일 경우 `0x00` 패딩 추가) 등을 주의 깊게 구현해야 합니다.

## 2. 트랜잭션 직렬화 (Transaction Serialization)

트랜잭션은 여러 구성 요소로 이루어진 복잡한 구조체입니다.

-   **기본 구조**:
    -   `version` (4바이트, 리틀 엔디안)
    -   `tx_inputs` (가변 길이 배열)
    -   `tx_outputs` (가변 길이 배열)
    -   `locktime` (4바이트, 리틀 엔디안)
-   **검토 사항**:
    -   **엔디안(Endianness)**: 대부분의 필드가 **리틀 엔디안(Little-Endian)**입니다. 현재 `intToBytes32`는 빅 엔디안이므로, 리틀 엔디안 변환 함수를 새로 만들거나 라이브러리를 사용해야 합니다.
    -   **가변 정수 (Variable-length Integer, VarInt)**: 배열의 길이나 스크립트 길이를 나타내는 데 사용됩니다. 1~9바이트로 크기가 변하므로 별도의 파싱/직렬화 로직이 필요합니다.
    -   **입력(TxIn)과 출력(TxOut)**:
        -   `TxIn`: 이전 트랜잭션 해시, 출력 인덱스, `scriptSig`(잠금 해제 스크립트), `sequence`로 구성됩니다.
        -   `TxOut`: 전송할 비트코인 양(사토시 단위), `scriptPubKey`(잠금 스크립트)로 구성됩니다.
    -   **스크립트**: 지금 단계에서는 `scriptSig`와 `scriptPubKey`를 단순히 가변 길이의 바이트 배열로 취급하면 됩니다. 스크립트 언어 자체의 파싱은 더 큰 작업이므로 나중에 다루어도 좋습니다.

## 3. 블록 직렬화 (Block Serialization)

블록은 헤더와 트랜잭션 목록으로 구성됩니다.

-   **블록 헤더 (80 바이트)**:
    -   `version`, `prev_block_hash`, `merkle_root`, `timestamp`, `bits` (난이도 목표), `nonce`로 구성됩니다.
-   **검토 사항**:
    -   트랜잭션과 마찬가지로 대부분 리틀 엔디안을 사용하며, 해시 값들은 내부적으로는 바이트 그대로지만 보통 16진수로 표시할 때는 순서를 뒤집어서 표현하는 점에 유의해야 합니다.
-   **머클 루트 (Merkle Root)**:
    -   블록에 포함된 모든 트랜잭션 해시들을 이진 트리 형태로 반복적으로 `hash256`하여 계산한 최종 해시 값입니다. 이 계산 로직을 새로 구현해야 합니다.

## 추천하는 Haskell 라이브러리

-   **`Data.Binary` (`binary` 패키지)**: 직렬화/역직렬화 로직을 안전하고 선언적으로 작성하는 데 매우 유용합니다. `impl-btc.cabal` 파일에 의존성을 추가하여 사용하는 것을 강력히 추천합니다. 엔디안 처리, 다양한 데이터 타입 지원 등이 강력합니다.

이 계획을 바탕으로 먼저 새로운 Haskell `data` 타입들을 정의하고, 이어서 각 타입에 대한 직렬화/역직렬화 로직을 구현해 나가시는 것을 권장합니다.

## 4. 네트워킹 (Networking)

비트코인은 P2P(Peer-to-Peer) 네트워크를 통해 통신합니다. 노드를 구현하려면 다른 노드와 연결하고, 메시지를 주고받는 방법을 알아야 합니다.

-   **피어(Peer) 탐색 및 연결**:
    -   **DNS 시드(Seed)**: 처음 네트워크에 참여할 때, 하드코딩된 몇 개의 DNS 주소에 질의하여 현재 활동 중인 노드들의 IP 주소 목록을 받아옵니다.
    -   **`addr` 메시지**: 일단 하나 이상의 노드에 연결되면, `getaddr` 메시지를 보내 더 많은 피어 주소를 요청하고 `addr` 메시지로 응답받아 자신의 피어 목록을 확장해 나갑니다.

-   **메시지 구조**: 모든 통신은 표준화된 메시지 형식을 따릅니다.
    -   **메시지 헤더**:
        -   `magic` (4바이트): 어떤 비트코인 네트워크인지를 식별하는 값 (메인넷, 테스트넷 등)
        -   `command` (12바이트): 메시지 종류를 나타내는 문자열 (예: `version`, `verack`, `getdata`)
        -   `length` (4바이트): 뒤따라오는 페이로드의 크기
        -   `checksum` (4바이트): 페이로드의 `hash256` 체크섬으로, 데이터 무결성을 검사하는 데 사용됩니다.
    -   **페이로드(Payload)**: `command`에 따른 실제 데이터. 이 부분이 바로 이전에 논의했던 직렬화된 블록, 트랜잭션 등이 포함되는 곳입니다.

-   **초기 연결 절차 (Handshake)**:
    1.  노드 A가 노드 B에게 자신의 프로토콜 버전 정보 등을 담은 `version` 메시지를 보냅니다.
    2.  노드 B는 `version` 메시지를 받고, 호환되면 `verack` (version acknowledgment) 메시지로 응답하며, 동시에 자신의 `version` 메시지를 노드 A에게 보냅니다.
    3.  노드 A도 노드 B의 `version` 메시지를 받고 `verack`으로 응답합니다.
    4.  양쪽 모두 `verack`을 주고받으면 핸드셰이크가 완료되고 데이터 교환을 시작할 수 있습니다.

-   **데이터 교환**:
    -   `inv` (Inventory): 자신이 가진 새로운 블록이나 트랜잭션의 해시 목록을 다른 노드에게 알릴 때 사용합니다.
    -   `getdata`: `inv` 메시지를 보고 자신이 갖지 않은 데이터가 있음을 알게 되면, 해당 해시를 담아 `getdata` 메시지로 전체 데이터를 요청합니다.
    -   `block` 또는 `tx`: `getdata` 요청에 대한 응답으로, 직렬화된 블록 또는 트랜잭션 데이터를 전송합니다.

## 5. 블록/트랜잭션 버전 관리 및 업그레이드

비트코인 프로토콜은 고정되어 있지 않고, 소프트 포크(Soft Fork)를 통해 계속 업그레이드되어 왔습니다. 따라서 최신 네트워크와 호환되려면 이러한 변경 사항들을 지원해야 합니다.

-   **Segregated Witness (SegWit)**: **가장 중요한 업그레이드**입니다. 현재 메인넷의 대다수 트랜잭션이 SegWit 형식을 따르므로 반드시 지원해야 합니다.
    -   **목표**: 트랜잭션 변조 가능성(malleability) 해결 및 블록 용량 증대.
    -   **작동 방식**: 서명 데이터(`scriptSig`)를 'witness'라는 별도의 구조로 분리합니다.
    -   **새로운 직렬화 형식**: SegWit 트랜잭션은 레거시 트랜잭션과 다른 직렬화 형식을 가집니다. `marker`(0x00)와 `flag`(0x01) 필드가 추가되고, 트랜잭션의 마지막에 모든 witness 데이터가 포함됩니다.
    -   **하위 호환성**: 구형 노드는 SegWit 트랜잭션을 "누구나 사용할 수 있는(anyone-can-spend)" 유효한 트랜잭션으로 인식하도록 설계되었습니다.
    -   **시사점**: 트랜잭션 직렬화/역직렬화 로직은 **레거시 형식과 SegWit 형식을 모두 구분하고 처리**할 수 있어야 합니다.

-   **Taproot (탭루트)**: 가장 최근의 주요 업그레이드입니다 (BIP-340, 341, 342).
    -   **목표**: 프라이버시, 효율성, 스크립트 기능 향상.
    -   **슈노르 서명(Schnorr Signatures)**: 기존 ECDSA를 대체하는 새로운 서명 방식을 도입했습니다.
    -   **P2TR(Pay-to-Taproot)**: 다중서명 등 복잡한 계약을 일반적인 단일 서명 거래처럼 보이게 하여 프라이버시를 크게 향상시키는 새로운 유형의 출력(output)입니다.
    -   **시사점**: 완전한 호환성을 위해서는 슈노르 서명과 P2TR을 지원해야 합니다. 초기 구현에서는 범위 밖일 수 있지만, 이러한 존재를 인지하고 파싱 로직을 설계하는 것이 중요합니다.

-   **트랜잭션 및 블록 버전 필드 (`nVersion`)**:
    -   이 필드는 단순히 버전을 나타낼 뿐만 아니라, `BIP-9`와 같은 제안에 따라 새로운 규칙(소프트 포크)에 대한 채굴자들의 지지 의사를 표현하는 비트 플래그로 사용되어 왔습니다. SegWit 활성화도 이 메커니즘을 통해 이루어졌습니다.
    -   초기 구현에서는 이 값을 파싱하는 것만으로도 충분하지만, 완전한 노드는 이 버전 비트를 해석하여 특정 블록이나 트랜잭션에 어떤 규칙을 적용할지 결정해야 합니다.
